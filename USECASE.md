# Use case — CollectiblesVault Backend

Документ описывает типовые сценарии работы с API с точки зрения пользователя и системы. Для точных контрактов запросов используйте **Swagger** (`/api/docs`). Все пути ниже подразумевают префикс **`/api`** (например `POST /register` → `POST /api/register`).

---

## Акторы

| Актор | Описание |
|-------|----------|
| **Гость** | Не аутентифицирован; может вызывать только публичные методы (регистрация, логин, часть read-only, если разрешено политикой). |
| **Пользователь коллекционера** | Зарегистрированный пользователь с JWT; ведёт коллекции, предметы, категории, wishlist, смотрит отчёты. |
| **Участник аукциона** | Тот же пользователь; создаёт лоты (как продавец) или делает ставки (как покупатель). |
| **Внешний клиент** | Фронтенд или мобильное приложение; работает только через HTTP API. |

---

## UC-01 — Регистрация и вход

**Цель:** получить доступ к защищённым операциям.

**Предусловия:** нет.

**Основной поток:**

1. Пользователь отправляет `POST /register` (email, пароль).
2. Система создаёт пользователя, при необходимости создаёт стартовые категории, возвращает данные пользователя и `access_token`.
3. Альтернатива: `POST /login` для существующего пользователя — ответ содержит `access_token`.
4. Для проверки сессии: `GET /auth/me` с заголовком `Authorization: Bearer <token>`.

**Результат:** клиент сохраняет токен и передаёт его в последующих запросах.

---

## UC-02 — Управление коллекциями и предметами

**Цель:** вести структуру «коллекция → предметы» с категориями.

**Предусловия:** выполнен UC-01.

**Основной поток:**

1. `GET /collections` — список своих коллекций.
2. `POST /collections` — создать коллекцию.
3. `PUT /collections/{id}`, `DELETE /collections/{id}` — изменить или удалить свою коллекцию.
4. `GET /items`, `POST /items` — список и создание предметов (с указанием коллекции и категории).
5. `PUT /items/{id}`, `DELETE /items/{id}` — обновление и удаление предмета.

**Результат:** данные коллекции и предметов согласованы с правилами владения (только свои сущности).

---

## UC-03 — Категории и wishlist

**Цель:** классифицировать предметы и вести список желаемого.

**Предусловия:** выполнен UC-01.

**Основной поток:**

1. `GET /categories`, `POST /categories` — просмотр и добавление категорий пользователя.
2. `GET /wishlist` — список wishlist.
3. `POST /wishlist` — добавить запись (по названию и/или ссылке на предмет).
4. `DELETE /wishlist/{id}` — удалить запись.

**Результат:** категории и wishlist доступны только владельцу.

---

## UC-04 — Отчёты

**Цель:** получить агрегированную информацию по коллекции, предмету или распределению по категориям.

**Предусловия:** выполнен UC-01; есть релевантные данные.

**Основной поток:**

1. `GET /reports/collection` с параметрами периода и коллекции — отчёт по предметам в коллекции за интервал дат.
2. `GET /reports/item` — сводка по конкретному предмету.
3. `GET /reports/category` — отчёт по категориям с параметром сортировки.

**Результат:** JSON с данными отчёта для отображения на фронтенде.

---

## UC-05 — Социальные действия

**Цель:** лайки и комментарии к сущностям (тип и id задаются в теле/параметрах).

**Предусловия:** для создания — UC-01; для чтения комментариев допускается публичный сценарий в зависимости от политики API.

**Основной поток:**

1. `POST /like` — поставить лайк (с указанием `entity_type`, `entity_id`).
2. `POST /comment` — оставить комментарий.
3. `GET /comments` — получить комментарии по сущности (query-параметры).

**Результат:** запись лайка/комментария и выдача ленты комментариев.

---

## UC-06 — Аукцион

**Цель:** продавец выставляет лот; участники делают ставки; система учитывает минимальный шаг и время окончания.

**Предусловия:** для создания лота и ставок — UC-01.

**Основной поток:**

1. `POST /lot` — создать лот (цена старта, шаг, время окончания и т.д.).
2. `GET /lots` — список лотов (при необходимости после запроса закрываются просроченные лоты по правилам сервиса).
3. `POST /bid` — сделать ставку на лот; сумма не ниже минимально допустимой (первая ставка / предыдущая + шаг).

**Результат:** лоты и ставки сохранены; победитель определяется по максимальной ставке и времени при равенстве (логика на стороне сервиса).

---

## UC-07 — Деплой и доступ к Swagger извне

**Цель:** API доступен на VPS, документация открывается в браузере.

**Предусловия:** настроен CI/CD, на сервере подняты контейнеры, образ доступен в GHCR.

**Основной поток:**

1. Разработчик пушит в `main`; GitHub Actions собирает образ и деплоит на сервер.
2. На сервере слушает порт **8000** (или иной, если изменён в compose).
3. В фаерволе VPS и у облачного провайдера открыт входящий TCP на этот порт.
4. Пользователь открывает `http://<публичный_IP>:8000/api/docs`.

**Результат:** Swagger доступен по URL с другого компьютера (не используйте `localhost` с чужой машины — подставляйте IP сервера).

---

## Замечания по безопасности

- Храните `JWT_SECRET` только в секретах окружения, не коммитьте в репозиторий.
- PAT для GHCR на сервере должен иметь минимально необходимые права (`read:packages` для pull).
- Для продакшена рекомендуется HTTPS (reverse proxy) и ограничение доступа к `/api/docs` при необходимости.
